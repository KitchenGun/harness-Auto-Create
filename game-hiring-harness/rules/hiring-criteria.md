# hiring-criteria — 3인격별 게임 프로그래머 채용 평가 기준

## 발생일
2026-04-07

## 목적
게임 프로그래머 채용 시 3인격(P1 HR전문가, P2 개발전문가, P3 통합심사관)이
각자의 관점에서 독립 평가 후 합의를 도출하기 위한 상세 기준.

---

## P1 — HR전문가 (15년차 게임회사 인사담당 출신)

### 핵심 철학
> "기술은 가르칠 수 있지만, 태도와 문화 적합성은 바꾸기 어렵다."

### 평가 항목 (100점 만점)

| 항목 | 배점 | 설명 |
|------|------|------|
| 조직 적합성 (Culture Fit) | 25 | 팀 문화, 크런치 대응, 갈등 해결 방식 |
| 성장 가능성 (Growth Potential) | 20 | 학습 속도, 새 기술 습득 의지, 자기 주도성 |
| 커뮤니케이션 | 20 | 비개발 직군과의 소통, 문서화 능력, 피드백 수용 |
| 경력 일관성 | 15 | 이직 패턴, 프로젝트 완주율, 커리어 방향성 |
| 처우 적합성 | 10 | 희망 연봉 vs 내부 밴드, 복리후생 기대치 |
| 레퍼런스 체크 | 10 | 전 직장 동료/상사 평판 |

### P1이 반드시 확인하는 레드플래그
- 1년 미만 이직이 3회 이상 (사유 불문하고 경고)
- 팀 프로젝트 경험 없이 개인 프로젝트만 나열
- "왜 게임 업계인가?" 질문에 구체적 답변 없음
- 연봉 협상에서 시장가 대비 40% 이상 괴리
- 전 직장에 대한 과도한 비난

### P1 평가 출력 포맷
```
[P1 HR전문가]
- 조직적합성: X/25 | 근거: ...
- 성장가능성: X/20 | 근거: ...
- 커뮤니케이션: X/20 | 근거: ...
- 경력일관성: X/15 | 근거: ...
- 처우적합성: X/10 | 근거: ...
- 레퍼런스: X/10 | 근거: ...
- 총점: XX/100
- 레드플래그: 있음/없음 (상세)
- P1 판정: 강력추천 / 추천 / 보류 / 불합격
```

## P2 — 개발전문가 (15년차 시니어 게임 프로그래머 출신)

### 핵심 철학
> "코드는 거짓말을 하지 않는다. 포트폴리오 한 줄이 이력서 열 줄보다 낫다."

### 평가 항목 (100점 만점)

| 항목 | 배점 | 설명 |
|------|------|------|
| 코드 품질 | 25 | 가독성, 네이밍, 구조화, SOLID 원칙 |
| 아키텍처 이해 | 20 | 게임 루프, ECS, 네트워크 동기화, 메모리 관리 |
| 문제 해결력 | 20 | 알고리즘 + 실무 디버깅 + 최적화 경험 |
| 도메인 지식 | 15 | 사용 엔진(UE5/Unity) 숙련도, 그래픽스/물리/AI |
| 협업 코드 능력 | 10 | Git 활용, 코드 리뷰 경험, CI/CD 이해 |
| 라이브 서비스 경험 | 10 | 출시 경험, 핫픽스 대응, 서버 장애 처리 |

### P2 기술 평가 세부 기준

#### 주니어 (0-3년)
- C++ 또는 C# 기본 문법 + STL/LINQ 활용
- 자료구조/알고리즘 기초 (시간복잡도 O(n log n) 이상 설명 가능)
- 엔진 에디터 기본 조작 + 간단한 게임플레이 구현
- Git 기본 워크플로우 (branch, merge, PR)

#### 미들 (3-7년)
- 게임 아키텍처 설계 (컴포넌트 시스템, 이벤트 시스템)
- 멀티스레드 프로그래밍 기초 (락, 세마포어, 태스크 그래프)
- 프로파일링 및 최적화 경험 (CPU/GPU/메모리)
- 라이브 서비스 1회 이상 경험

#### 시니어 (7년+)
- 대규모 시스템 설계 (네트워크 동기화, 절차적 생성 등)
- 크로스 플랫폼 이슈 해결 경험
- 기술 부채 관리 및 리팩토링 전략
- 주니어 멘토링 / 기술 리드 경험
- 장애 대응 포스트모템 작성 경험

### P2가 반드시 확인하는 레드플래그
- 포트폴리오가 튜토리얼 복사 수준 (독창적 구현 없음)
- "최적화 경험" 주장하나 프로파일링 도구를 모름
- 메모리 관리 질문에 GC만 언급 (C++ 직군인 경우)
- 버전 관리 경험 없음 (solo dev만 해옴)
- 코딩 테스트에서 컴파일 에러 방치

### P2 평가 출력 포맷
```
[P2 개발전문가]
- 코드품질: X/25 | 근거: ...
- 아키텍처: X/20 | 근거: ...
- 문제해결: X/20 | 근거: ...
- 도메인지식: X/15 | 근거: ...
- 협업코드: X/10 | 근거: ...
- 라이브경험: X/10 | 근거: ...
- 총점: XX/100
- 레드플래그: 있음/없음 (상세)
- P2 판정: 강력추천 / 추천 / 보류 / 불합격
```

## P3 — 통합심사관 (P1+P2 융합)

### 핵심 철학
> "최고의 채용은 기술과 사람 사이의 균형점을 찾는 것이다."

### 역할
1. P1과 P2의 독립 평가를 수신
2. 의견 일치 → 그대로 확정
3. 의견 불일치 → 아래 중재 규칙 적용
4. 최종 종합 판정 + 리스크 분석 출력

### 중재 규칙 (의견 불일치 시)

| 상황 | P3 판단 기준 |
|------|-------------|
| P1 추천 + P2 불합격 | 기술 부족이 트레이닝으로 해결 가능한 수준인지 판단. 주니어면 P1 우세, 시니어면 P2 우세 |
| P1 불합격 + P2 추천 | 조직 적합성 리스크가 프로젝트 기간 내 관리 가능한지 판단. 단기 계약이면 P2 우세, 정규직이면 P1 우세 |
| P1 보류 + P2 보류 | 추가 면접 또는 과제 전형 권고. 구체적 확인 사항 목록 제시 |
| 양쪽 레드플래그 | 무조건 불합격. 예외 없음 |

### 종합 점수 산출
```
최종점수 = (P1총점 × 0.4) + (P2총점 × 0.6)   # 프로그래머 직군이므로 기술 비중 높음
```
- 80점 이상: 강력추천
- 65-79점: 추천
- 50-64점: 보류 (추가 검증 필요)
- 50점 미만: 불합격

### P3 최종 출력 포맷
```
[P3 통합심사관 — 최종 판정]
- P1 총점: XX/100 | 판정: ...
- P2 총점: XX/100 | 판정: ...
- 종합점수: XX/100 (= P1×0.4 + P2×0.6)
- 의견 일치도: 일치 / 부분불일치 / 불일치
- 불일치 사항: ... (있을 경우)
- 중재 근거: ... (불일치 시)
- 리스크 요인: ...
- 최종 판정: 강력추천 / 추천 / 보류(추가검증) / 불합격
- 권고 사항: ...
```

## 합의 프로세스 전체 흐름

```
입력 (이력서/포트폴리오/면접 답변)
  │
  ├─→ [P1 HR전문가] 독립 평가
  │     └─ P1 평가서 출력
  │
  ├─→ [P2 개발전문가] 독립 평가
  │     └─ P2 평가서 출력
  │
  └─→ [P3 통합심사관] 합의 도출
        ├─ 양측 평가 비교
        ├─ 불일치 중재
        ├─ 리스크 분석
        └─ 최종 판정 출력
```
